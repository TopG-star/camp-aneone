# FIN-001a: Bank Statement Intake Foundation

## Objective

Create a safe, idempotent foundation for finance ingestion by detecting candidate bank-statement
emails and storing intake records for later parsing.

## Scope

### In Scope

1. Sender allowlist + subject keyword candidate detection.
2. Durable intake record storage in `bank_statements`.
3. Idempotent duplicate protection on `(user_id, source, external_id)`.
4. Intake status tracking: `discovered`, `skipped_duplicate`, `queued_for_parse`.
5. Immutable intake evidence fields:
   - `message_id`
   - `thread_id`
   - sender
   - subject
   - `received_at`
   - `detection_rule_version`

### Out of Scope

1. Attachment fetching/downloading.
2. PDF parsing and statement normalization.
3. Transaction extraction and categorization.
4. Conversational finance tools.
5. Finance dashboard pages, charts, and aggregates.

## Acceptance Criteria

1. Candidate detection can be configured without code changes.
2. Re-polling the same message does not create duplicate rows.
3. Intake records preserve immutable message evidence for auditability.
4. Intake status transitions are deterministic and test-covered.
5. Existing ingestion flows (non-finance) remain behaviorally unchanged.

## Delivery Boundary

FIN-001a is considered complete when data-level intake behavior is stable and verified.
Any parsing, analytics, chat, or dashboard work requires a follow-up scope approval (FIN-001b+).

## Verification Checklist

1. Unit tests cover candidate detection and duplicate handling.
2. Integration tests validate unique-key idempotency in SQLite.
3. Application/infrastructure typecheck passes after changes.
4. No PRD scope contradictions remain.
